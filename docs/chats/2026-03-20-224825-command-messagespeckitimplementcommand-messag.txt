‚Źļ How the speckit workflow works

  Everything lives in specs/001-daily-journal-app/:

  spec.md          ‚Üź WHAT to build (user stories, requirements)
    ‚Üď
  plan.md          ‚Üź HOW to build it (architecture, tech stack, file structure)
    ‚Üď
  data-model.md    ‚Üź Database entities and relationships
  contracts/       ‚Üź Service interface definitions
  research.md      ‚Üź Tech decisions and tradeoffs
  quickstart.md    ‚Üź Setup instructions
    ‚Üď
  tasks.md         ‚Üź ORDERED work items derived from the plan

  The tasks.md file is where we're tracking progress ‚ÄĒ each [x] is done, each [ 
  ] is remaining. That's the source of truth.

  How pivoting works

  Depends on what changes:

  Small tweak (e.g., "change the accent color", "add a field to entries"):
  ‚Üí Just edit the code directly. Update tasks.md if relevant.

  Modify a feature (e.g., "emotion tracking should use a wheel instead of a
  grid"):
  ‚Üí Run /speckit.specify to update spec.md with the new requirement, then
  /speckit.tasks to regenerate affected tasks. Completed tasks stay done.

  Add a new feature (e.g., "I want tagging/categories for entries"):
  ‚Üí Update spec.md ‚Üí re-run /speckit.plan ‚Üí /speckit.tasks. New tasks get
  appended.

  Remove a feature (e.g., "drop voice dictation"):
  ‚Üí Update spec.md, mark those tasks as N/A or delete them, remove the code.

  Major pivot (e.g., "actually make this a web app"):
  ‚Üí Re-run the full pipeline: /speckit.specify ‚Üí /speckit.plan ‚Üí /speckit.tasks.
   Start fresh or salvage what applies.

  The key idea: specs drive everything downstream. If you change spec.md, the
  plan and tasks should be regenerated to stay in sync. You don't have to ‚ÄĒ you
  can just edit tasks.md directly for small changes ‚ÄĒ but for bigger shifts the
  pipeline keeps things coherent.
